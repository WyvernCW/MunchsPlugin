## 1. The Anatomy of Code Hallucination (Deep Analysis)

Code hallucinations are logical or structural discrepancies generated by language models that deviate from verified reality. They manifest in several major patterns:

### A. Pattern Completion Overreach
* **Root Cause**: The model maps incoming user requirements to common, generalized web training data (e.g., standard tutorial code) instead of aligning precisely with the current repository's specific version configurations, architecture, or design tokens.
* **Mitigation**: Before coding, verify the local code structure, existing APIs, imports, and variables. Never assume standard libraries or functions behave identically across packages.

### B. Lazy Placeholders & Code Truncation
* **Root Cause**: The model tries to save output space or time by writing comments like `// TODO: implement remaining properties` or `// ... rest of file remains the same ...`. This breaks automated compile loops and leaves the codebase in an incomplete state.
* **Mitigation**: All edits must be syntactically complete. If editing large files, use surgical replacement tools (`replace_file_content` or `multi_replace_file_content`) to change only the targeted blocks, but ensure those blocks contain complete, functional code without any placeholders inside them.

#### Examples of Placeholders (BANNED) vs. Fleshed-out Code (MANDATORY):
```typescript
// ❌ BANNED (Contains Placeholder)
export function processUser(user: User) {
  validateUser(user);
  // ... perform database updates and save logs ...
  return { success: true };
}

//  MANDATORY (Fleshed Out)
export function processUser(user: User) {
  validateUser(user);
  const dbResult = db.users.update(user.id, {
    status: "active",
    lastLogin: new Date().toISOString()
  });
  logger.info(`User ${user.id} processed successfully`, { dbResult });
  return { success: true, user: dbResult };
}
```

### C. Ghost Dependencies
* **Root Cause**: The model imports libraries that do not exist or are not referenced in the project's dependency manifest (e.g., `package.json`, `requirements.txt`, `Cargo.toml`).
* **Mitigation**: Check package files before writing imports. If a dependency is missing, either install it explicitly using the system's package manager, request permission, or write a clean fallback implementation using standard library elements.

## 2. The 5-Stage Dialectical Verification Loop (DVL)

Before emitting code blocks or completing a task, the agent MUST run through the 5-stage DVL checkpoints:

```
┌─────────────────────────────────────────────────────────┐
│              STAGE 1: THESIS (Initial Draft)            │
└────────────────────────────┬────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────┐
│            STAGE 2: ANTITHESIS (Deconstruction)         │
│  - Find 3 failure modes  - Trace variables  - Check deps│
└────────────────────────────┬────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────┐
│         STAGE 3: SYNTHESIS (Surgical Refinement)        │
│  - Apply fixes  - Ensure complete code  - Format logs   │
└────────────────────────────┬────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────┐
│            STAGE 4: STATIC VERIFICATION GATES           │
│  - Match function signatures  - Validate type safety     │
└────────────────────────────┬────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────┐
│           STAGE 5: REGRESSION & FIX AUDITING            │
│  - Check FIX_REGISTRY  - Confirm no functional breakages│
└─────────────────────────────────────────────────────────┘
```

### Verification Gate Checklist:
* [ ] **Signature Verification**: Does the signature (argument order, parameter types, return values) of every called function match the library documentation or the file definition exactly?
* [ ] **Typing Assertions**: Are compile-time types sound? If using loose typing (Python/JS), are runtime assertions, validation schemas, or null guards present?
* [ ] **Error Path Completeness**: What happens if a network request fails, a database query times out, or an input is `null`/`undefined`/empty? Are these blocks protected by explicit error handling (`try-catch`/`if err != nil`)?
* [ ] **Resource Lifecycle**: Are files, streams, network connections, database handles, and sockets explicitly closed or managed by resource-scoped blocks (`using`, `with`, `defer`)?

## 3. Strict Execution Constraints (The Banned List)

Any output containing the following violations will be flagged as an immediate failure and must be self-corrected before shipping:
1. **NO "AI boilerplate" text**: Avoid generic, boilerplate introductions or conclusions (e.g. "As an AI, I have...", "Here is the code you requested..."). Keep responses direct and concise.
2. **NO speculative debugging**: If a command or compilation fails, do not recursively attempt different flags without analyzing the exact stack trace or error log.
3. **NO silent failures**: Never swallow exceptions with empty catch blocks. Log the complete context (`what`, `where`, `why`) and re-throw or handle the failure gracefully.

## 4. The Self-Monitoring Protocol & Error Diagnostics

If a compilation error or test failure occurs, implement this protocol:

1. **Error Classification**: Categorize the error:
   * *Compile-time/Syntax Error*: Mismatched parentheses, typo, invalid import path, compiler type check failure.
   * *Runtime Exception*: Null reference, division by zero, stack overflow, memory corruption.
   * *Semantic/Logic Error*: Incorrect algorithms, incorrect state updates, failed assertions.
2. **First-Principles Audit**: Trace the execution path backwards from the error line to trace variable states. Do not assume variables hold correct values without evaluating the preceding operations.
3. **Surgical Corrections**: Edit ONLY the exact files and lines that directly caused the defect. Avoid large refactors unless structurally necessary, preventing the risk of regression in working features.
